## OSI-PDU
### 1. PDU 的结构：数据 + 控制信息

你可以把 PDU 想象成一个**快递包裹**：
- **服务数据单元 (SDU)**：包裹里的“货”，即来自上层的数据。
- **协议控制信息 (PCI)**：包裹上的“快递单”，包含目标地址、纠错码、协议版本等。

$$PDU = SDU + PCI$$
虽然统称为 PDU，但在不同的层级，为了方便区分，工程师们给它们起了不同的名字。这是网络工程面试或考证（如 CCNA）中的必考点：

| **层级**                | **PDU 的名称**                          | **助记说明**               |
| --------------------- | ------------------------------------ | ---------------------- |
| **应用层 / 会话层 / 表示层**   | **数据 (Data)**                        | 原始的消息内容。               |
| **传输层 (Transport)**   | **段 (Segment)** / **报文段** / **数据段 ** | 将大数据拆分成一小段一小段，贴上端口号。   |
| **网络层 (Network)**     | **包 (Packet)** / **分组**              | 贴上源 IP 和目的 IP，准备跨网络路由。 |
| **数据链路层 (Data Link)** | **帧 (Frame)**                        | 贴上 MAC 地址，并在末尾加上校验信息。  |
| **物理层 (Physical)**    | **比特 (Bit)**                         | 纯粹的 0 和 1 电信号或光脉冲。     |
### 2. PDU 的封装与解封装过程

这个过程就像俄罗斯套娃：

1. **封装 (Encapsulation)**：当你发送信息（如在 Android App 里点击“发送”）时，数据从应用层向下走。每一层都给它套一个“壳”。比如网络层套上 IP 头，它就变成了一个 **Packet**。

2. **解封装 (De-encapsulation)**：当对方手机收到电信号（Bit）后，会一层层拆开。数据链路层拆掉 MAC 头看一眼是不是给自己的，网络层拆掉 IP 头看地址对不对。

**关键点**：由于“对等层通信”的存在，接收方的传输层只看发送方传输层封装的那个 **Segment** 头部，而不会去管下层是怎么包的。
### 例子
#### 1. 应用层 (Application) —— 产生“数据 (Data)”

你在浏览器地址栏输入域名，浏览器会构建一个 **HTTP 请求**（或者进行 DNS 查询）。

- **PDU 内容**：一段文本，比如 `GET /index.html HTTP/1.1`。
- **逻辑对话**：你的浏览器在跟北大服务器的 Web 软件说话。
- **添加信息**：为你输入的域名添加了一个http超文本传输协议的请求头

#### 2. 传输层 (Transport) —— 封装成“分段 (Segment)”

由于 HTTP 数据可能很大，传输层（通常是 TCP 协议）会把它切成小块。

- **添加信息**：源端口号（你的浏览器随机分配）和目的端口号（HTTP 默认是 **80**）。
- **PDU 名称**：**TCP Segment**。
- **逻辑对话**：确保这些数据块按顺序到达，没有丢失。

#### 3. 网络层 (Network) —— 封装成“包 (Packet)”

这一层负责给数据找路。

- **添加信息**：你的 IP 地址（源）和北大服务器的 IP 地址（目的）。
- **PDU 名称**：**IP Packet**。
- **逻辑对话**：路由器通过查看这个 Packet 的头部，决定下一步把它发往哪个城市。

#### 4. 数据链路层 (Data Link) —— 封装成“帧 (Frame)”

数据准备在物理线路上跑了，但它需要知道下一跳的具体硬件地址。

- **添加信息**：你电脑网卡的 **MAC 地址** 和路由器（网关）的 **MAC 地址**。
- **PDU 名称**：**Ethernet Frame**。
- **逻辑对话**：确保数据能准确地从你的电脑传到寝室的路由器。

#### 5. 物理层 (Physical) —— 转换成“比特 (Bit)”

- **处理**：所有的头部和数据都被转换成电压的高低、光脉冲或无线电波。
- **PDU 名称**：**Bits**。
- **逻辑对话**：网线另一头接收到这些 0 和 1。

---
#### 总结：接收方的逆向操作

当北大的服务器收到这些 **Bits** 后，会发生以下“拆快递”行为：

1. **物理层**：把电信号变回 **Bits**。
2. **数据链路层**：拆掉 **Frame** 头部，检查 MAC 地址对不对。
3. **网络层**：拆掉 **Packet** 头部，检查 IP 地址，发现是找自己的。
4. **传输层**：拆掉 **Segment** 头部，发现这是给 80 端口（Web 服务）的数据。
5. **应用层**：终于拿到了原始的 **HTTP 请求**，服务器知道你要看主页了！



## 实际应用：TCP/IP协议
### TCP/IP 的四层架构

TCP/IP 把 OSI 的七层精简成了 **4 层**（有时为了教学方便也会拆分为 5 层）。

|**TCP/IP 层级**|**对应 OSI 层级**|**职责**|**常见的协议**|
|---|---|---|---|
|**应用层** (Application)|应用层、表示层、会话层|处理特定的应用程序细节。|HTTP, DNS, FTP, MQTT|
|**传输层** (Transport)|传输层|为两台主机提供端到端通信。|**TCP**, **UDP**|
|**网络层** (Internet)|网络层|处理分组在网络中的活动（找路）。|**IP**, ICMP|
|**网络接口层** (Link)|数据链路层、物理层|处理与电缆/光纤等物理介质的交互。|以太网, Wi-Fi|
 >[!IMPORTANT]
 >TCP/IP架构 **不等于** TCP协议！

## TCP与UDP的比较

 在网络世界里，**TCP (Transmission Control Protocol)** 就像是一个**极致负责的顺丰快递员**，而 **UDP (User Datagram Protocol)** 就像是一个**只管把信塞进邮筒的投递员**。
 
---
### 1. 核心差异对比表

|**特性**|**TCP（传输控制协议）**|**UDP（用户数据报协议）**|
|---|---|---|
|**连接性**|**面向连接**（必须先握手）|**无连接**（直接发数据）|
|**可靠性**|**可靠传输**（丢包重传、顺序纠正）|**不可靠传输**（发出去就不管了）|
|**传输方式**|**字节流**（像自来水一样连续）|**报文**（像一个个独立的包裹）|
|**速度/延迟**|慢（因为有确认机制和拥塞控制）|**快**（没有握手和复杂的逻辑）|
|**资源消耗**|高（需要维护连接状态、缓存）|**低**（非常轻量）|
|**通信方式**|一对一|一对一、一对多（广播/组播）|

---

### 2. 深度拆解：TCP 到底在忙什么？

TCP 之所以被称为“可靠”，是因为它在后台为你做了大量你看不见的工作：

- **三次握手（建立连接）**：
    在发数据前，双方必须确认彼此的收发能力。就像打电话：“喂，听得到吗？”“听到了，你能听到我吗？”“我也听到了。”

- **确认应答 (ACK)**：
    你每发一个包，对方都要回一个“收到”。如果没收到回信，TCP 就会认为包丢了，自动重发。

- **流量控制 & 拥塞控制**：
    如果对方处理不过来（手机太卡），或者网络很拥堵，TCP 会自动**减慢发送速度**。它非常具有“公德心”，不会把网络彻底堵死。

- **数据排序**：
    如果你发了 1、2、3 三个包，但 3 先到了，TCP 会把 3 存在缓冲区，等 1 和 2 到了再按顺序交给你的 App。

---

### 3. 深度拆解：UDP 为什么这么“简单”？

UDP 几乎不对数据做任何修饰。它只是在你的应用层数据前贴了一个 8 字节的简易头部（包含源端口和目的端口）。

- **没有握手**：想发就发，不需要等待对方响应。
- **没有重传**：丢了就丢了，UDP 不会尝试修复。
- **没有拥塞控制**：它会以你能提供的最高速度猛冲，不管网络受不受得了。

---

### 4. 形象类比：看电影 vs 视频通话

- **TCP 就像“在线看电影（优酷/Netflix）”**：
	你不能容忍画面少了一帧或者台词跳了行。如果网络不好，你可以忍受加载圈转几秒（缓冲），但出来的画面必须是完整的、顺序正确的。

- **UDP 就像“视频通话（微信视频/Zoom）”**：
    实时性是第一位的。如果网络抖动，画面花一下、声音卡一秒没关系，只要能接着聊就行。你绝对不希望看到 10 秒前的画面现在才重传回来，“画面回溯”在通话里是灾难。

---

### 5. 在你的项目里怎么选？

作为一名 PKU 的开发者，你在做决策时可以参考这个逻辑：

1. **选 TCP 的场景**：
    - **英语学习 App 的单词本同步**：绝对不能丢数据，必须准确。
    - **登录、支付**：安全性、准确性第一。
    - **文件传输、网页浏览**：数据必须完整。

2. **选 UDP 的场景**：
    - **在线多人游戏**：比如《原神》或《王者荣耀》的位置同步，快比准重要。
    - **语音/视频流**：直播、VoIP。
    - **物联网 (IoT) 传感器**：很多传感器每秒发好几次温度，偶尔丢一个没关系，省电和低延迟更重要。
