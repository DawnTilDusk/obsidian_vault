## 一、 宏观心法：编程的“面”

这部分讲的是程序员对待代码应有的基本态度和架构思维。

- **代码不是写完就结束了，要反复推敲：** 代码行数不是衡量水平的标准 。优秀的程序员会不断回头提炼、反思和重构自己的代码，也就是所谓的“代码审查（Code Review）” 。

- **拒绝“面条代码”，追求结构化：** 优雅的代码在结构上不应该像面条一样绕来绕去、多层嵌套 。它应该像分门别类的小盒子，或者枝丫分明的树状结构，逻辑清晰 。

- **真正的模块化是逻辑隔离，而非物理隔离：** 把代码拆到不同的文件目录里不叫模块化 。真正的模块化是指拥有“定义良好的输入和输出”，**函数**本身就是最好的模块化工具 。

- **克制炫技，追求直观与简单：** 不要盲从语言的各种新“特性”和奇技淫巧（教员甚至建议尽量不用 C++11 及以后的复杂语法） 。如果有直接、清晰的写法，哪怕显得稍微长一点、笨一点，也要优先选择 。

## 二、 微观习惯：写代码的“点”

课件中列举了大量具体的编码坏习惯及其改进方案，可以直接应用到你的日常开发中。

## 1. 基础规范

- **全英文环境：** 变量名、注释、甚至 Github 上的交流，尽量都使用英文，避免在 Linux 环境下出现乱码 。

## 2. 函数的设计规范

- **保持短小：** 建议函数长度不超过 40 行（刚好是一个屏幕能显示的最大行数），不需要滚动鼠标就能看完，这样大脑能完整处理这段逻辑 。

- **单一职责原则：** **每个函数只做一件简单的事情** 。如果一个函数内部要根据条件去“选择”做不同的事，说明它应该被拆分。换句话说，每个函数应该只有一个被修改的理由 。

- **抽取辅助函数：** 不要害怕函数调用的开销（现代编译器会自动 inline）。发现重复代码，或者一段很长且复杂的逻辑（即使只用一次），都应该提取成带名字的小函数，用函数名代替注释 

- **少用类成员传参：** 不要为了图省事，用类的成员变量作为两个函数之间传递数据的“通道”。这会破坏函数的独立性，应该尽量使用局部变量和函数参数传递信息 。

## 3. 变量的使用与命名

- **好名字胜过注释：** 用有意义的函数名和变量名来表达逻辑，这是最顶级的可读性 。

- **局部变量要“近”：** 局部变量应该在尽量靠近它被使用的地方定义，不要习惯性地全堆在函数的开头 。

- **上下文清晰时，名字要短：** 局部变量名字应该尽量简短。比如在 `if` 语句中，用 `success` 就比用 `successInDeleteFile` 好，因为结合上下文读者已经知道你在做什么了 。

- **绝对不要“重用”变量：** 为了节省变量名，反复对同一个变量赋不同意义的值是极其糟糕的习惯。不同的概念就应该用不同的变量名 。

- **提取复杂的中间表达式：** 遇到长串的计算或逻辑判断，不要全挤在一行，应该将其拆解，赋值给有意义的中间变量 。


## 4. 逻辑控制与排版

- **永远不要省略花括号 `{}`：** 即使 `if` 或 `for` 后面只有一行代码，也必须写 `{}`。省略花括号极易引起“光学幻觉”（看着像是在条件内，实际在条件外），导致严重的 Bug 。

- **长条件合理换行：** 当 `if` 里的条件太长时，必须进行换行和对齐排版，让逻辑一目了然 。

- **别用短路逻辑代替流程控制：** 虽然 Shell 或 C++ 支持 `&&` 和 `||` 的短路特性，但**不要**用它来替代 `if-else` 执行复杂的动作流，那样会非常难以阅读 。

- **凡事必想 `else`：** 写代码要做到无懈可击，当写下 `if` 时，必须强迫自己思考条件不成立时系统应该做什么操作，错误处理要前置 。
