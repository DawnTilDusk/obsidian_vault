## 一、 `< >` 与 `" "` 的区别：去哪儿找文件？

编译器在看到 `#include` 时，就像是在电脑里玩“捉迷藏”。
#### 1. `< >` （系统路径优先）

- **查找逻辑**：编译器直接去**标准库**或**系统指定目录**（比如 `/usr/include` 或安装编译环境时设定的文件夹）里找。
- **适用场景**：调用 C++ 标准库（如 `<iostream>`, `<vector>`）或第三方库头文件。
- **特点**：效率高，不会搜索你的工程目录。
#### 2. `" "` （当前路径优先）

- **查找逻辑**：编译器先在**当前源文件所在的目录**（你自己的工程文件夹）找。如果没找到，再去系统路径找。
- **适用场景**：调用你自己写的头文件（如 `"MessageBoard.h"`）。
- **特点**：范围广，但如果你用它包含标准库（如 `#include "iostream"`），编译器会先在你的文件夹里白忙活一圈，虽然最后能找到，但浪费了时间。

## 二、 后缀名的区别：文件是干什么的？

在 C++ 开发中，不同的后缀决定了文件在项目架构中的“身份”。

|**后缀名**|**全称**|**主要用途**|**核心特点**|
|---|---|---|---|
|**`.h`**|Header|**声明** (Declaration)|传统的 C 风格头文件。存放类定义、函数声明、宏定义。|
|**`.hpp`**|Header Plus Plus|**声明 + 部分实现**|专门为 C++ 设计。常用于 **模板类 (Template)**，因为模板必须在头文件里实现。|
|**`.cpp`**|C Plus Plus|**实现** (Implementation)|C++ 的源文件。存放具体的函数逻辑、变量赋值。|
|**`.c`**|C|**C 语言源文件**|纯 C 语言的代码。C++ 编译器通常也能跑 `.c` 文件，但不建议混用。|

## 三、 为什么要有这种拆分？（底层逻辑）

你可能会问：**为什么不把所有代码都写在一个文件里？**

1. **编译速度**：
    如果你改了 `.cpp` 里的一个逻辑，编译器只需要重新编译这一个文件。但如果你把实现都写在 `.h` 里（且被很多文件引用），改一个地方，整个项目都要重新编译。
    
2. **避免重复定义**：
    如果在头文件里直接写函数实现（如 `int add(int a, int b) { return a+b; }`），当两个 `.cpp` 同时包含这个头文件时，链接器（Linker）会报错：“兄弟，`add` 函数被定义了两次，我听谁的？”
    
3. **隐藏细节**：
    你给别人提供库时，只需要给 `.h` 文件（告诉他怎么用）和编译好的二进制文件，不需要把 `.cpp` 源码给他。

## 组织代码块

你可以按照这种“职责分配”来组织你的代码块：

- **`MessageBoard.h` (或 .hpp)**:
```cpp
    class MessageBoard {
        static int total_boards; // 声明
        void print();            // 声明
    };
```

- **`MessageBoard.cpp`**:
```cpp
    #include "MessageBoard.h"
    int MessageBoard::total_boards = 0; // 定义/初始化
    void MessageBoard::print() { ... }  // 具体实现
    ```
