## 1. OpenMP 核心概念：执行与内存模型

### 进程 vs 线程

- **进程 (Process)**：操作系统资源分配的基本单位。拥有独立的内存资源，进程间通信（如 MPI）需要显式发送/接收消息，开销大 。

- **线程 (Thread)**：调度执行的基本单位，存在于进程内部。**OpenMP 使用多线程**，不同线程之间**共享内存**，可以直接读写共享变量来通信，创建和调度的开销比进程小得多 。

### Fork-Join 执行模式

OpenMP 程序始终从一个**主线程 (Master thread)** 开始 。

1. **Fork（分叉）**：当主线程遇到并行区指令（如 `#pragma omp parallel`）时，会派生出一组子线程 。

2. **并行执行**：所有线程并发执行并行区内的代码 。

3. **Join（合并）**：并行区结束处有一个隐式同步点，所有线程在此等待。完成后只有主线程继续往下执行串行代码 。

## 2. 线程与环境控制 (编程三要素)

OpenMP 编程主要依赖三要素：**编程指导语句 (Pragmas)**、**库函数 (API)**、**环境变量** 。

### 决定线程数的优先级 (从低到高)

并行区中的线程数按以下顺序决定 ：

1. **系统默认**：一般为机器可用的 CPU 逻辑核数 。
2. **环境变量**：如在终端设置 `export OMP_NUM_THREADS=8` 。
3. **库函数**：在代码中调用 `omp_set_num_threads(n)` 。
4. **从句**：指令后加 `num_threads(n)` 。
5. **条件并行**：`if(条件)`，满足条件才开启多线程 。

**代码示例：基础环境控制与 `if` 从句**

```c++
#include <iostream>
#include <omp.h> // 必须引入头文件 [cite: 122]

int main() {
    int nthreads = 4;
    omp_set_num_threads(nthreads); // 库函数设置线程数 [cite: 410]
    
    // 条件并行：只有 nthreads > 3 时才开启多线程 [cite: 411]
    #pragma omp parallel if(nthreads > 3) 
    {
        int id = omp_get_thread_num(); 
        // 库函数：获取当前线程的 ID (从0开始) [cite: 322]
        std::cout << "Hello from thread " << id << std::endl;
    }
    return 0;
}
```

## 3. 工作共享与并行循环构造

SPMD（单程序多数据）模式虽然灵活，但手动通过线程 ID 计算数据边界非常繁琐 。OpenMP 提供了更强大的**共享工作循环构造**来自动分配任务 。

### `#pragma omp parallel for`

- 这是最常用的组合指令，用于将 `for` 循环的迭代任务自动平分给多个线程同时执行 。

- **注意**：紧跟指令的循环控制变量（如 `int i`）会被自动私有化，每个线程拥有自己的副本，避免冲突 。

### 规约操作：`reduction(op:var)`

- **问题**：当多个线程累加同一个共享变量（如 `sum += i`）时，会产生数据竞争 。

- **原理**：OpenMP 为每个线程创建一个 `var` 的私有副本（例如加法初始化为 0，乘法为 1）。各线程先在私有副本上计算，循环结束后再通过指定的二元操作（`+`, `*`, `max`, `min`）将所有私有副本的值合并到全局原始变量中 。

**代码示例：计算数组和的规约操作**

```c++
#pragma omp parallel for reduction(+:sum) // [cite: 814]
for (int i = 0; i < n; ++i) {
    sum += array[i]; // 各线程安全累加，无须加锁 [cite: 816]
}
```

### 循环合并：`collapse(n)`

- **原理**：将 `n` 层嵌套循环展平合并为一个单一的大循环，相当于增加了外层循环的迭代次数，这有助于线程调度器更均匀地分配任务 。

- **约束**：被合并的多重循环之间**必须没有任何数据依赖关系** 。

## 4. 同步机制 (解决数据竞争与时序问题)

并发执行意味着指令执行顺序不可预知 。当多个线程同时访问同一个内存区域，且至少有一个是写操作时，就会发生**数据竞争**，导致结果错误（如随机输出）。

### 互斥同步 (保护共享资源)

互斥锁通过强制操作串行化来保证数据完整性，每次只允许一个线程访问保护区 。

- **`critical` (临界区)**：
    - 作用于**代码块**。在任一时刻，最多只有一个线程能进入该代码块执行 。

- **`atomic` (原子操作)**：
    - 是一个微型的 `critical`，只对**一条特定的表达式**（如 `sum += i`）起作用 。
    - **优势**：编译器底层优化更好，系统开销更小，执行速度比 `critical` 更快 。能用 `atomic` 的地方尽量用 。

```c++
#pragma omp parallel for
for(int i = 0; i < 10000; ++i) {
    #pragma omp atomic // 高效保护单条语句 [cite: 1156]
    sum += i; [cite: 1158]
}
```

### 事件同步 (控制执行流)

- **隐式 Barrier (屏障)**：在 `parallel`, `for`, `single` 等构造的结尾，默认有一个隐式的同步屏障，所有线程必须在此等待，直到全部到达才能继续 。

- **`nowait`**：加在 `for` 循环指令末尾（如 `#pragma omp for nowait`），用于**取消隐式 Barrier**。执行完分配任务的线程可以无需等待，直接执行后面的代码 。(注意：`nowait` 不能和 `parallel for` 组合使用 )。

- **`#pragma omp barrier`**：显式插入一个同步屏障。所有线程必须同时到达此行代码后，才能继续往下执行 。

### 特殊线程控制

- **`master` 构造**：代码块**仅由主线程**（ID 为 0）执行。**结尾没有隐式 Barrier**，其它线程不会等待，会直接跳过并继续执行后面的语句 。

- **`single` 构造**：代码块由到达此处的**任意单个线程**执行（不一定是主线程）。**结尾有隐式 Barrier**，其余线程会在此等待执行的线程完成 。

- **`ordered` 构造**：强制在循环中标记的代码区严格按照 `for` 循环中 `i` 的先后顺序串行执行（性能开销较大）。

## 5. 实用性能测试工具

- **`omp_get_wtime()`**：OpenMP 提供的精度极高的库函数，用于获取当前时间（以秒为单位）。通常通过前后两次调用的差值来计算并行程序的运行耗时 。

```c++
double t0 = omp_get_wtime(); [cite: 923]
// ... 执行并行代码 ...
double t1 = omp_get_wtime(); [cite: 938]
cout << "Time elapsed: " << t1 - t0 << " seconds." << endl; [cite: 941]
```
