整个链路是这样的（反向追踪）：
1. 浏览器 (main.js) : fetch('/api/latest') -> 找 app.py 要数据。
2. 后端 (app.py) : 查询数据库 SELECT * FROM sensor_data ... -> 把存在库里的数据拿出来。
3. 中转站 (udp_relay.py) : 监听 UDP 端口 -> 接收采集器发来的数据包 -> 写入数据库。
4. 采集器 (C 程序) : 读取硬件传感器 -> 打包成 UDP 数据包 -> 发送给 udp_relay.py 。


专有名词解释：
### 1. UDP (User Datagram Protocol)
- 全称 : User Datagram Protocol (用户数据报协议)。
- 是什么 : 一种  **“发完不管”** 的网络传输方式。
  - 特点 : 速度快，但不保证对方一定收到（就像寄平信，丢了也不负责）。
  - 项目中的作用 : 你的采集器（C语言程序）用 UDP 拼命往外发温度数据。因为它发得很快（比如一秒一次），丢一两个包无所谓，所以选 UDP 最合适，效率最高。
### 2. SQL (Structured Query Language)
- 全称 : Structured Query Language (结构化查询语言)。
- 是什么 : 人类和数据库沟通的 通用语言 。
- 特点 : 就像英语是国际通用语一样，不管你用什么数据库软件（openGauss, MySQL, PostgreSQL），都用 SQL 来指挥它。
- 项目中的作用 :
  - SELECT * FROM sensor_data : "嘿，把所有传感器数据拿给我看看。"
  - INSERT INTO sensor_data ... : "给我在表里记上一笔新数据。"
### 3. Templates (模板)
- 全称 : Web Templates (网页模板)。
- 是什么 : 带有 **“填空题”** 的 HTML 文件。
- 特点 : 它不是写死的静态页面，而是留了很多坑（占位符）。
- 项目中的作用 : templates/index.html 就是一个模板。
  - 它是这样的： \<h1>当前温度：{{ temp }}\</h1>
  - 当 Flask ( app.py ) 渲染它时，会把 {{ temp }} 替换成真实的 25.5°C ，然后发给浏览器。这样就不需要为每个温度都写一个新网页了。
### 4. DB (Database)
- 全称 : Database (数据库)。
- 是什么 : 电子化的文件柜 。
- 特点 : 专门用来存数据，存取速度极快，而且支持复杂的查找。
- 项目中的作用 : 你的 sensor_data 表就是数据库的一部分。它把所有零散的温度、光照数据整整齐齐地存起来，让你以后能查到“上个月的今天温度是多少”。

### 5. Flask 在项目中的三大核心作用：
**A. 充当“前台接待” (路由/Routing)**
Flask 决定了当用户访问某个网址时，应该执行什么 Python 代码。
在 app.py 中，你看到很多 @app.route(...) 这样的装饰器，这就是在分配任务。

- 例子 ：
  ```
  @app.route('/')
  def index():
      return render_template('index.html')
  ``` 
  解释 ：当用户在浏览器输入 http://开发板IP:5000/ 时，Flask 说：“哦，你要去首页（ / ）”，于是它调用 index() 函数，把 index.html 页面展示给你。 
  
**B. 充当“传菜员” (API 接口)**
前端网页（ main.js ）需要最新的温度、光照数据，它不能直接去读数据库或传感器，必须问 Flask 要。

- 例子 （你之前问的）：
  ```
  @app.route('/api/latest')
  def api_latest():
      # ... 去数据库查最新数据 ...
      return jsonify(latest_data)
  ``` 
  解释 ：当 main.js 发起 fetch('/api/latest') 请求时，Flask 收到请求，去后台查好数据，打包成 JSON 格式（数据包裹），然后返回给前端。
  
**C. 充当“装修工” (模板渲染/Templates)**
虽然 HTML 决定了网页的结构，但有时候我们需要在网页发出去之前，先把一些数据填进去。Flask 使用 Jinja2 模板引擎来做这件事。

- 例子 ： render_template('index.html') 解释 ：Flask 读取 templates/index.html 文件，如果里面有 {{ variable }} 这样的占位符，Flask 会把 Python 变量的值填进去，生成最终的 HTML 给用户。


# 温度模块的技术链路
按道理来说如果udp数据包里面的temp_c是NULL，那么payload里面也是NULL，于是notify_backend会broadcast一个NULLtemp，然后yield一个NULL温度值，最后在onmessage函数里面调用updateLatestDisplay函数，那么按道理来说NULL应该会被过滤掉吧

你的推导逻辑非常严密， 你是对的 ，前端确实会“过滤”掉（即不更新）这个 NULL 温度值。

我们来一步步验证这个“过滤”发生在哪里：

1. UDP Relay ( udp_relay.py ) :
   
   - payload = {"temperature": None, ...}
   - 发送给后端。
2. 后端 Flask ( app.py:api_relay_notify ) :
   
   ```
   t = data.get('temperature')  # t 
   是 None
   # ...
   try:
       valid_temp = (t is not None) 
       and (-40 < float(t) < 125)
   except Exception:
       valid_temp = False
   
   if valid_temp:
       cur['temperature'] = float(t)
   # 注意：这里没有 else 分支去删除 cur
   ['temperature'] 或者把它设为 None！
   ```
   - 关键点 ： cur 是 LATEST_CACHE 的引用（或者副本）。如果 LATEST_CACHE 里 之前已经有了一个有效的温度值 （比如 25.5），而这次 t 是 None ，那么 cur['temperature'] 不会被更新 ，它依然保持着上次的值（25.5）。
   - 如果 LATEST_CACHE 是全新的（刚启动）， cur 里可能根本没有 temperature 字段。
3. SSE 推送 :
   
   - broadcast(cur) 发送的是 cur 这个字典。这个字典里要么保留着旧的温度值，要么根本没有温度字段（如果从未收到过有效温度）。 它绝不会包含 temperature: None 。
4. 前端 ( main.js:updateLatestDisplay ) :
   
   ```
   if (tempElement && Object.
   prototype.hasOwnProperty.call
   (data, 'temperature')) {
       const t = data.temperature;
       if (typeof t === 
       'number' && ...) {
           // 更新 UI
       }
   }
   ```
   - 由于后端的逻辑， data.temperature 要么是一个有效的数字（旧值或新值），要么该字段不存在。
   - 即使后端发来了 temperature: null （假设逻辑改了），前端的 typeof t === 'number' 检查也会拦截它（因为 typeof null 是 'object' ）。
### 结论
是的，前端不会显示 NULL 或者 0 。

- 如果这次数据包里温度是 NULL，后端会 直接忽略 这次温度更新，继续向前端推送 上一次缓存的有效温度 。
- 所以前端看到的温度数值不会变成 "--" 或 "0"，而是保持不变，直到下一次有效的温度数据到来。
这解释了为什么虽然数据库里存了一堆 0.0，但你在网页上看到的实时温度并没有疯狂跳变到 0 度。因为 实时推送的数据源（LATEST_CACHE）和数据库存储逻辑是分离的 。数据库存了脏数据，但实时缓存因为校验逻辑的存在，反而避开了脏数据。





### 方案一：维持两个独立采集器（进程）
现状 ： sensor_temp_collector （读 AHT10）和 sensor_light_collector （读光敏）分别运行。

- 优点 ：
  
  1. 解耦与容错 ：这是最大的优势。如果光敏传感器坏了或者 I2C 通信卡死了导致进程崩溃， 温度采集完全不受影响 。反之亦然。在竞赛或实际工程中，这种“互不拖累”的特性非常宝贵。
  2. 代码清晰 ：每个程序只专注做一件事（Single Responsibility Principle），代码短小精悍，易于维护和调试。
  3. 灵活性 ：你可以为温度设置 1Hz 采样率，为光敏设置 0.5Hz 采样率，互不干扰。
- 缺点 ：
  
  1. I2C 总线竞争 ：两个进程同时尝试操作同一个 I2C 设备文件（ /dev/i2c-7 ），虽然 Linux 内核的 I2C 驱动支持多进程访问（内核会自动排队），但如果访问频率极高，理论上会有微小的上下文切换开销（在嵌入式 Linux 上通常可以忽略）。
  2. 资源占用 ：多一个进程就多一份内存和 CPU 调度开销（同样，对于现代板子来说微乎其微）。
### 方案二：合并为一个超级采集器
构想 ：写一个 sensor_env_collector ，在一个 while(1) 循环里依次读 AHT10 和 光敏。

- 优点 ：
  
  1. 数据同步 ：你可以确保发出的每个数据包里，温度和光照是 同一时刻 采集的。这对于某些需要严格时序关联的数据分析（比如分析“光照引起温度升高”的滞后性）很有用。
  2. 总线控制 ：单进程完全掌握 I2C 总线的使用权，你可以精确控制读完 AHT10 后休眠多少毫秒再读光敏，绝对不会发生冲突。
- 缺点 ：
  
  1. 单点故障 ：如果读取光敏传感器的代码因为 I2C 错误卡死（比如 read 阻塞），或者发生了段错误， 整个进程挂掉，温度数据也没了 。
  2. 采样率耦合 ：如果你想让温度 1秒一变，光照 5秒一变，代码逻辑会变得复杂（需要写计数器或非阻塞定时器）。
