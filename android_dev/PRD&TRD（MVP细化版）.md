## 一、 全局架构与技术基调 (MVP 细化版)

### **目标用户**：
中小学生（UI 需具备大色块、高对比度、圆角卡片等适龄特征，交互需直观防误触、且带有轻量级的物理动效反馈）。

### **核心IP**：Garden，森林配色
- **核心色盘 (Core Palette)**：
	- **主色 (Primary):** `#66BB6A` (生机绿) —— 用于高亮 Tab、核心进度条、签到成功态。
	- **辅助色 (Secondary):** `#8D6E63` (橡木棕) —— 用于次级文字、泥土元素、未选中的 Tab 线框。
	- **点缀色 (Accent):** `#FFB74D` (暖阳橙) —— 用于金币/代币数值、重要成就、新功能红点。
- **空间底色 (Surfaces)**：
	- **全屏背景 (Background):** `#F1F8E9` (淡薄荷绿) —— 护眼低饱和底色，替代纯白，营造森林氧气感。
	- **卡片容器 (Surface):** `#FFFFFF` (奶油白) —— 配合 **24dp 大圆角**与 **8% 透明度的绿色软阴影**，增加呼吸感。
- **状态表现 (UI States)**：
	- **热力图渐变:** 从 `#E8F5E9` (浅苔) 平滑过渡至 `#2E7D32` (深林)。
	- **交互动态:** 翻页指示器在拉长时使用主色，且边缘带有微小的透明度渐变，模拟阳光穿透树叶的虚化效果。
**设计准则**：拒绝高频对比，坚持“大圆角+低饱和+自然光影”，确保长时间英语学习下的视觉舒适度。

### 核心布局与导航架构 (双层联动架构)**：
- 应用摒弃传统的静态 Tab 切换，采用 **“横向滑动翻页器（Horizontal Pager)** 作为主体的容器。四大核心业务模块（Tab 1-4）作为 Pager 的四个页面，支持用户通过手指在屏幕中央左右滑动来平滑切换。

- **中控联动层 (指示器/插槽)**：在主内容区与底部导航栏之间，悬浮一层分页指示器。MVP 阶段默认为 4 个小圆点。当页面滑动时，当前激活页对应的圆点会具有**“拉长（胶囊状）并加深颜色”**的动态过渡效果。这里在代码层保留“插槽（Slot API）”设计，未来可以无缝替换为带有 Garden IP 元素的微型卡片（例如滑动到特定页，小圆点变成一朵含苞待放的花）。

- **底层常驻层 (底部导航栏)**：最底部为常驻的四大 Tab 图标。它与中控联动层共享同一个页面状态（State）。随着页面的滑动，对应的 Tab 图标会平滑过渡到高亮状态；同时，点击任意 Tab 图标，不仅会高亮，还会反向驱动上方的页面容器执行“平滑滚动（Smooth Scroll）”到对应页面的动画。

### **全局路由**：
进入主界面的 Pager 容器前，通过一层独立的启动/签到页面（Splash/Check-in）进行前置路由拦截，完成每日仪式感后再初始化主框架。

### 闭环注意
- 为了支撑上述的双层联动导航，并在未来完美支撑“学习产生数据 -> 数据浇灌花园 -> 花园产出成就”的全局闭环，你在搭建底层代码结构时，**必须**在全局层面预留以下几个核心接口和状态管理器：

#### 1. 全局单向数据流控制器 (Global App State / Source of Truth)

由于四大页面现在是可以随时滑动切换的，如果 Tab 2 正在背单词，滑动到 Tab 3 时，数据必须是实时互通的。

- **预留接口**：需要一个全局的 `UserSessionRepository`。
    
- **作用**：它统管用户的“当前代币（Tokens）”、“今日总学习时长（StudyTime）”和“当前词汇量（VocabularySize）”。所有 Tab 只能**订阅（Observe）**这个库的数据变化，或者通过明确的意图（Intent/Action）去请求修改它，确保跨页面滑动时数据绝对同步，不会出现状态撕裂。
    

#### 2. 页面容器与双导航的“状态同步锁” (Pager State Synchronizer)

滑动页面、指示器变形、Tab 高亮，这三者必须完美同频。

- **预留接口**：在 Scaffold 最顶层抽取一个提升后的状态 `val pagerState = rememberPagerState(...)`。
    
- **作用**：将其向下传递给“页面内容区”、“圆点指示器”和“底部导航栏”。这样，无论是滑动屏幕触发了 `pagerState.currentPage` 的改变，还是点击底部 Tab 触发了 `pagerState.animateScrollToPage()`，所有的 UI 组件都能基于同一个数据源做出反应。
    

#### 3. 全局奖励/动效事件总线 (Global Reward Event Bus)

考虑到交互的趣味性，MVP 阶段就需要为跨组件动效留好后路。

- **预留接口**：一个热数据流（如 `SharedFlow<RewardEvent>`）。
- **作用**：假设用户在 Tab 2 完成了一个任务，触发了一个掉落金币的动作。即使用户立刻滑动到了 Tab 4，系统也能通过这个总线接收到事件，并在 Tab 4 的界面上渲染出“金币入账”的 UI 动效。

#### 4. 可扩展的插槽化 UI 接口 (Slot-based UI Architecture)

针对你提到的“之后绘制一些和花园相关的卡片替换小圆点”。

- **预留接口**：底部导航上方的指示器不要写死为 `DotIndicator()`。应该定义一个带有 `@Composable` 参数的容器，比如 `CustomIndicatorPanel(content: @Composable () -> Unit)`。
    
- **作用**：在 MVP 阶段，我们往这个插槽里传入普通的拉长小圆点；到了 V2.0 阶段，只需传入一个新的 `GardenSeedIndicator()` 组件即可，完全不需要改动底层的滑动和联动逻辑。

## 二、 页面详细拆解与组件规划

## 1. 启动与签到页 (Splash & Check-in)

- **UI 布局**：
    - 全屏显示的“背景壁纸” 。建议使用贴合“Garden”主题的精美插画（如清晨的花园）。
    - 页面中央有醒目的“签到”按钮 ，并带有一个大号的勾选图标（✅）。

- **交互与逻辑**：
    - 用户每日首次打开应用时强制展示，强化仪式感。
    - 点击“签到”后，触发金币/代币掉落动画，随后路由跳转至首页。

- **技术映射**：可使用 `Box` 布局实现背景图铺底，上层叠加居中的 `Button` 或 `Surface`。

## 2.  Tab 1：首页概览 (Dashboard/Home)

#### UI 布局

- **左侧 - 学习热力图 (Heatmap/Calendar)**：
    - **标题层**：包含“热力图”和“Calendar”文本标识 。
    - **矩阵层**：一个基于网格（Grid）的打卡展示区，模拟 GitHub 贡献图 。
    - **MVP 预留**：在网格组件外层封装一个 `Box` 或 `Surface`，为未来可能加入的“花园背景剪影”预留层级空间。

- **右侧 - 每日任务列表 (Daily Mission)**：
    - **标题层**：包含“每日任务”标识 。
    - **列表层**：由多个任务卡片组成，每个卡片包含任务描述及状态占位符（如数字 0 或复选框） 。
    - **MVP 预留**：任务卡片的 Data Class 需预留一个 `rewardType` 字段，方便后续对接“种子”、“水滴”或“代币”等 Garden 专属奖励。

- **底部导航栏 (Navigation Bar)**：
    - 固定在屏幕底部，明确显示当前处于“首页”激活状态 。

#### 交互与逻辑

- **热力图动态渲染**：
    - **核心逻辑**：系统需读取历史打卡数据（Check-in Data），根据完成度动态映射网格颜色深浅 。
    - **MVP 必须注意**：建立一个独立的 `StatisticsRepository`。即便现在只存“是否打卡”，也要支持未来扩展“学习时长”、“正确率”等权重数据，因为这些将直接影响热力图的颜色深度。

- **任务状态实时更新**：
    - **核心逻辑**：用户在列表中勾选或完成任务后，UI 必须通过响应式数据流（如 StateFlow）立即回馈 。
    - **MVP 必须注意**：引入“事件总线”或“状态统一管理”机制。当 Tab 2 的学习模块完成后，必须能跨页面通知 Tab 1 的任务列表刷新。

#### 技术映射

- **热力图组件**：使用 `LazyVerticalGrid` 实现。建议在 MVP 阶段就将每个“小方块”抽离为独立组件，方便后续在方块内加入微小的植物图标
- **任务列表组件**：使用 `LazyColumn` 实现 。
- **数据流驱动**：采用 `ViewModel` 暴露 `State`，确保热力图和任务列表共用同一个时间维度的 Source of Truth。

#### 易扩展性

1. **统一奖励回调接口 (Reward Observer)**： 在任务列表的 `onTaskCompleted` 逻辑里，预留一个 `onRewardTriggered(reward)` 的空方法。现在它什么都不做，但以后这就是 Garden 里的花“突然长大”的触发开关。
2. **日期感知型架构 (Date-Aware Logic)**： 热力图和任务列表必须强绑定系统时间。建议在 MVP 阶段就写好“跨天重置”逻辑，即当用户在第二天打开应用时，任务列表能自动刷新，而不是停留在昨天的完成状态。
3. **Slot API 设计 (针对 Compose)**： 在编写 Tab 1 的布局组件时，多用 `content: @Composable () -> Unit` 这种 Slot 方式。这样以后你想在热力图旁边塞一个小装饰，或者在任务列表头上加个进度条，不需要重写整个布局，直接往“槽位”里塞东西就行。
## 3. Tab 2：学习模块中心 (Learning Hub)
- **UI 布局**：
	- **网格容器 (Grid Container)**：
	    - 采用 **2 列多行** 的响应式网格布局。
	    - 卡片包含：**背单词**、**语法**、**词汇测验**、**教材训练**、**听力训练**、**写作/专项**。
	- **标准功能卡片 (Module Card)**：
	    - **视觉元素**：每个卡片预留“图标（Icon）”+“主标题（Title）”+“微描述（Subtitle）”的结构。
	    - **MVP 预留**：在卡片右上角预留一个“红点/进度”的小位置（Badge），方便后续接入“今日未完成”或“新内容”提示。
	- **底部导航栏**：
	    - 焦点切换至“学习”图标。

- **交互与逻辑**：
	- - **统一路由分发**：
	    - **核心逻辑**：点击卡片不应直接写死跳转逻辑，而是通过一个 `StudyRouter` 进行分发。
	    - **MVP 注意**：每个卡片点击时需传入一个 `moduleId`。这样做是为了方便在 Tab 1 的任务列表中实现“一键直达”功能。

	- **模块状态预读**：
	    - **核心逻辑**：进入页面时，异步加载各个模块的简要状态（例如：背单词模块显示“今日还剩 20 词”）。
	    - **MVP 注意**：即使现在不显示具体进度，架构上也要支持 `ModuleStatus` 的查询接口，避免后期为了加个进度条要重构整个 View。
	    - 考虑到是中小学生，卡片建议加入对应的拟物化或卡通图标（如单词卡对应一本小字典，听力对应小耳机）。

- **技术映射**：
	- **布局实现**：使用 Compose 的 `VerticalGrid`（固定列数为 2）。
	- **组件解耦**：将 6 个功能块抽象为一个通用的 `@Composable StudyModuleCard`。
	- **数据结构**：定义一个 `ModuleConfig` 类，包含 `title`, `icon`, `targetRoute` 等字段，通过 `List<ModuleConfig>` 循环渲染出整个页面。

- **易拓展性和闭环注意**
	-  **“空实现”占位逻辑 **
		- **建议**：对于还没开发的“听力”或“语法”，点击后弹出一个精美的 `Toast` 或 `SnackBar` 提示：“该功能正在 Garden 中萌芽...”，而不是让卡片点击没反应。这符合产品的“优雅降级”原则。

	-   **结果回调契约 (ActivityResult Contract)**
		- **逻辑**：从 Tab 2 进入“背单词”子页面，学习完成后返回时，子页面必须返回一个 `StudyResult` 对象。
		- **目的**：Tab 2 拿到结果后，立即上报给全局的 `TaskManager`。这样当你回到首页（Tab 1）时，你会发现任务列表已经自动勾选，热力图颜色也变深了。这个“闭环”必须在架构最开始就设计好。
## 4. Tab 3：数据与花园 (Data & Garden - Core IP)

- **UI 布局**：导航栏焦点切换至“数据” 。这是整个 APP 最具特色的页面。
    - - **左侧双层看板 (Stats Panel)**：
	    - **上方 - 环形比例图**：标题为“学习时间分布” ，用于展示不同学习模块（如单词 vs 听力）的时长占比。
	    - **下方 - 趋势折线图**：标题为“词汇量” ，展示随时间推移的词汇增长曲线。
	- **右侧养成区 (Garden View)**：
	    - **标题层**：包含“Garden”标识 。
	    - **网格土地 (Grid Plot)**：一个 4x4 或 5x5 的网格区域，用于放置植物组件 。
	    - **MVP 预留**：在网格上方预留一个悬浮的“天气/环境”层，未来可以根据学习积极性切换“晴天”或“雨天”。
	- **底部导航栏**：
	    - 焦点切换至“数据”图标 。

- **交互与逻辑**：
    - **数据驱动的“植物生长”**：
	    - **核心逻辑**：花园不直接读取学习记录，而是监听一个 `GardenEngine`。当 `StatisticsRepository` 更新（如词汇量增加）时，触发 `GardenEngine` 分发“养料”。
	    - **MVP 注意**：植物的生长状态（种子 -> 幼苗 -> 开花）应由简单的数值阈值驱动。例如：词汇量每增加 50，指定网格的植物 Level +1。
	- **图表交互响应**：
	    - **核心逻辑**：点击折线图上的点，应能弹出当天的详细数据小气泡。
	    - **MVP 注意**：图表应支持“周/月/年”切换的接口预留，虽然 MVP 阶段可能只展示周数据。

- **技术映射**
	- **自定义绘图 (Canvas)**：考虑到 Jetpack Compose 的特性，建议使用 `Canvas` 绘制环形图和折线图，这比引入第三方繁重的图表库更利于你练手 
	- **花园网格 (Garden Compose)**：建议将花园网格设计为一个独立的自定义 Layout。每个格点是一个 `Slot`，可以放置不同的 `@Composable Plant(level: Int)`。
	- **状态持久化**：使用 `Room` 存储 `GardenState`（哪个格点种了什么，当前几级），确保退出应用后花园不会重置。

- **易拓展性和闭环注意**
	- 1. “养料”转换器 (The Fertilizer Converter) —— 闭环核心
		这是连接学习与 IP 的关键。
		- **逻辑**：建立一个中间件，将 Tab 2 产生的“学习成果”转换为“花园能量”。
		- **扩展性**：今天 10 个单词 = 1 滴水，明天你可以轻松改为 10 个单词 = 1 块化肥，而不需要改动 UI 代码。
    

	-  2. 植物组件的策略模式 (Strategy Pattern)
		不要为每种花写一个类
		- **建议**：定义一个 `PlantSpecies` 接口。
		- **好处**：MVP 阶段你可能只有“小草”，但只要接口定好，以后增加“向日葵”或“仙人掌”只需要替换贴图资源，逻辑完全通用。
    
	- 3. 异步数据刷新 (State Reducer)
		由于 Tab 3 涉及复杂的图表计算和动画。
		- **逻辑**：使用 `collectAsStateWithLifecycle` 来监听数据。
		- **注意**：当用户在 Tab 2 完成任务并切回 Tab 3 时，数据应能平滑过渡（例如折线图有一个向上弹跳的动画），这种“即时反馈”是中小学生最喜欢的爽点。
## 5. Tab 4：个人中心 (Profile & Rewards)

- **UI 布局**：
    - **左侧 - 身份信息卡片 (Identity Card)**：
	    - **布局**：由两个垂直排列的圆角卡片组成 。
	    - **内容**：上方卡片显示头像、昵称、年级；下方卡片显示当前等级（LV）及升级进度条。

	- **右侧 - 奖励与资产 (Asset Gallery)**：
	    - **布局**：占据右侧较大空间的面板，标题为“成就/代币” 。
	    - **内容**：上方显示“数字代币”余额（Garden 专用货币）；下方是“成就勋章”展示墙，用于陈列学习里程碑。
	- **底部导航栏**：
	    - 焦点切换至“我的”图标。

- **交互与逻辑**
	- **代币同步逻辑 (Currency Sync)**：
	    - **核心逻辑**：实时监听全局资产数据库。当用户在 Tab 2 完成任务获得代币，或在 Tab 3 购买种子消耗代币时，此处余额需自动刷新。
	- **成就触发机制 (Achievement Trigger)**：
	    - **核心逻辑**：基于“事件监听”而非“手动检查”。例如，当词汇量（Tab 3 数据）达到 1000 时，系统自动向该页面推送“千词斩”勋章。
    - **MVP 注意**：成就系统应设计为“可插拔”的。勋章图标在未获得时显示灰度，获得后高亮并显示获取日期。

- **技术映射**
	- **数据持久化 (Room)**：使用数据库存储 `UserProfile` 和 `UserAssets` 表。
	- **状态管理 (Shared ViewModel)**：由于代币和成就可能被多个 Tab 影响，建议使用 `SharedViewModel` 或全局单例 `Repository` 来管理这些 State。
	- **图像加载**：成就勋章推荐使用 `Vector Asset`（矢量图）以减少包体积，并支持 Compose 的色调过滤（ColorFilter）轻松实现灰度/彩色切换。****

- **易拓展性和闭环注意**
	-  1. 资产路由接口 (Transaction API)
		不要分散地修改代币数值。
		- **建议**：建立一个统一的 `EconomyManager` 接口，提供 `addCoins(amount, reason)` 和 `spendCoins(amount, item)` 方法。
		- **扩展性**：这样你可以轻松记录每一笔“账单”（例如：_因背单词奖励 10 金币_），方便未来给家长看学习报告。

	- 2. “空位”占位符设计 (The Empty Slot Strategy)
		- **建议**：在成就墙预留几个带问号的“神秘成就”位置。
		- **目的**：给中小学生留悬念，引导他们去探索 Tab 2 的深度功能（例如连续打卡 30 天解锁神秘种子）。

	-  3. 闭环的核心：消费出口 (The Spending Loop)
		Tab 4 虽然是展示页，但它决定了 Tab 3（花园）的繁荣度。
		- **闭环注意**：在架构上，Tab 4 的“代币”是连接 Tab 2（生产）和 Tab 3（消费）的唯一媒介。
